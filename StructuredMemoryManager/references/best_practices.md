# 设计思路、优先级策略与归档原理

> 本文是 StructuredMemoryManager 的设计哲学深度解读，帮助理解每个设计决策背后的原因。

---

## 一、为什么需要结构化记忆？

### 1.1 默认记忆方式的问题

大多数 Agent 采用**按时间分文件**的记忆方式（如 `2026-06-24.md`），这种方式存在以下问题：

| 问题 | 描述 | 后果 |
|------|------|------|
| 信息分散 | 同类信息散落在多个日期文件中 | 检索时需遍历大量文件 |
| 无分类 | 习惯、技能、项目混杂在一起 | 无法针对性查找 |
| 无优先级 | 所有信息权重相同 | 重要约束可能被淹没 |
| 无过期机制 | 信息永不过期 | 过时信息干扰判断 |
| 索引缺失 | 只能全文搜索 | 效率低，噪音大 |

**StructuredMemoryManager 的解决方案**：按**类别**而非时间组织文件，每个条目以**内容摘要**为标题，便于直观理解和检索。

### 1.2 结构化方案的核心优势

```
┌─────────────────────────────────────────────┐
│              memory_index.md                │  ← 总目录（一级索引）
│  ┌──────────┬──────────┬──────────┐        │
│  │  习惯文件 │  技能文件 │  项目文件  │        │  ← 分类聚合
│  │ ┌──────┐ │ ┌──────┐ │ ┌──────┐ │        │
│  │ │索引0 │ │ │索引0 │ │ │决策0 │ │        │  ← 内部索引（二级索引）
│  │ │索引1 │ │ │索引1 │ │ │决策1 │ │        │
│  │ │...   │ │ │...   │ │ │...   │ │        │
│  │ └──────┘ │ └──────┘ │ └──────┘ │        │
│  └──────────┴──────────┴──────────┘        │
└─────────────────────────────────────────────┘
```

**三级检索**确保每次读取最少量的数据即可定位目标。

## 二、优先级策略详解

### 2.1 三级优先级的设计逻辑

优先级不仅仅是排序依据，更是**生存策略**：

```
high (核心约束)
  ├── 永不自动归档
  ├── 写入 priority_tags（启动时预加载）
  ├── 代表用户的硬性规则
  └── 示例："不要生成图片"、"必须用中文回复"

medium (常用偏好)
  ├── 正常保留，定期检查
  ├── 到期后提示确认
  ├── 代表稳定的个人偏好
  └── 示例："偏好暗色主题"、"用TypeScript"

low (临时信息)
  ├── 90天后纳入归档候选
  ├── 到期自动降权
  ├── 代表一次性或易变的信息
  └── 示例："本次任务用Tab缩进"、"临时约定"
```

### 2.2 优先级的动态调整机制

优先级不是静态的，可以通过 `confirm_memory` 工具调整：

```
新记忆写入 (default: medium)
       │
       ▼
   使用过程中发现很重要？
       │
       ├─ Yes → upgrade to high
       │         └→ 加入 priority_tags → 启动时预加载
       │
       └─ No → 保持 medium
              │
              ▼
         到期了 (expires)?
              │
              ├─ 仍有效? → confirm → extends forever
              │
              └─ 已过时? → 不操作 → auto downgrade to low
                              │
                              ▼
                         90天 old?
                              │
                              ├─ Yes → archive candidate
                              └─ No → 继续保留
```

### 2.3 priority_tags 的作用

`priority_tags` 是一个**全局可见的高优信号系统**：

1. **在 memory_index.md 中**：标注哪些文件包含核心约束
2. **在文件 front matter 中**：快速了解该文件最重要的内容
3. **在 Agent 启动时**：扫描所有 `high_priority_tags`，预加载对应条目到上下文

这确保了即使记忆量很大，核心约束也能在 O(1) 时间内被发现。

## 三、归档原理与实现

### 3.1 为什么需要归档？

随着使用时间增长，记忆文件会不断膨胀：

```
假设每天产生 2 条记忆：
  1个月  →  60 条   ← 接近阈值
  3个月  → 180 条   ← 大量过时信息
  1年    → 730 条   ← 检索效率显著下降
```

归档解决三个问题：
1. **性能**：保持活跃文件在合理大小内（≤50条）
2. **准确性**：减少过时信息对检索结果的干扰
3. **可追溯性**：归档不等于删除，历史记录仍然可查

### 3.2 归档算法详解

```python
def should_archive(entry, now):
    """
    判断条目是否应被归档
    必须同时满足两个条件：
    """
    condition_1 = (entry.priority == "low")      # 条件1：低优先级
    condition_2 = age_in_days(entry.date) > 90    # 条件2：超过90天

    return condition_1 and condition_2
```

**设计决策说明**：

| 选择 | 原因 |
|------|------|
| 只归档 low 优先级 | high 和 medium 的信息仍有价值 |
| 90天阈值 | 约3个月，足够长以避免误删，足够短以保持文件精简 |
| 不删除而是移动 | 保留完整历史供必要时检索 |
| 双阈值策略 | 第一轮90天未清理干净则降为60天 |

### 3.3 归档流程图

```

```

```
       │

```

## 四、跨文件关联的设计思想

### 4.1 为什么需要关联？

真实世界的知识不是孤立存在的：

```
项目A 需要 → 技能X（如何做Y）→ 用户偏好Z（风格要求）
```

没有关联机制时，这些联系只能靠 Agent "偶然想起"。关联让这种关系**显式化、可追踪**。

### 4.2 关联的类型

| 类型 | 示例 | 建立时机 |
|------|------|---------|
| 项目→技能 | 项目使用了某自动化脚本 | add_memory 时检测到 |
| 技能→项目 | 技能来源于某项目实践 | add_memory 时检测到 |
| 技能→技能 | 方法A是方法B的变体 | 手动或智能检测 |
| 习惯→项目 | 项目遵循特定编码规范 | add_memory 时检测到 |

### 4.3 关联的价值

1. **上下文扩展**：检索时提供"相关记忆您可能也想知道"
2. **归档保护**：被高优文件引用的 low 优先级条目可豁免归档
3. **依赖分析**：理解哪些记忆之间存在依赖关系

## 五、三级检索的性能优势

### 5.1 传统方式 vs 三级检索

假设共有 1000 条记忆，分布在 20 个文件中：

| 操作 | 传统全文搜索 | 三级检索 |
|------|-------------|---------|
| 读取数据量 | ~2MB（全部文件） | ~5KB（YAML索引） |
| I/O 次数 | 20次（每文件一次） | 2-4次 |
| 解析开销 | 全文正则匹配 | YAML结构化查询 |
| 结果精度 | 高噪音 | 高精准 |
| 延迟 | 高 | 低 |

### 5.2 各级别的职责

```
Level 1 - memory_index.md (~1KB)
  职责：文件级路由
  类比：图书馆楼层导航牌
  成本：极低（只需读一个文件）

Level 2 - internal_index (~5-10KB)
  职责：条目级定位
  类比：书架上的书籍目录卡片
  成本：低（只解析YAML，不读正文）

Level 3 - 正文条目 (~0.5-2KB)
  职责：内容获取
  类比：翻开书阅读具体章节
  成本：中（只读一个条目，非全文）
```

## 六、工具独占原则的设计理由

### 6.1 为什么禁止裸写文件？

如果允许 Agent 直接写记忆文件，会出现：

1. **索引不一致**：写了正文但忘了更新 internal_index
2. **格式破坏**：手动拼接 YAML 容易出错
3. **归档绕过**：直接写入导致条目计数不准，归档失效
4. **并发冲突**：多步操作中间被打断导致半更新状态

### 6.2 工具方法的保证

通过 `add_memory` / `search_memory` / `confirm_memory` / `rebuild_index` 四个工具：

```
原子性：单次调用完成全部更新
一致性：索引和正文始终同步
完整性：自动触发归档检查
可追溯：所有操作都有明确的输入输出
```
