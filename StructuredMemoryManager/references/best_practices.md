# 设计思路与最佳实践

> 本文档是 StructuredMemoryManager 的设计哲学深度解读，帮助理解每个设计决策背后的原因。

---

## 一、为什么需要结构化记忆？

### 1.1 默认记忆方式的问题

大多数 Agent 采用按时间分文件的记忆方式（如 `2026-06-24.md`），存在以下问题：

| 问题 | 后果 |
|------|------|
| 信息分散，同类信息散落在多个日期文件中 | 检索时需遍历大量文件 |
| 无分类，习惯/技能/项目混杂在一起 | 无法针对性查找 |
| 无优先级，所有信息权重相同 | 重要约束可能被淹没 |
| 无过期机制，信息永不过期 | 过时信息干扰判断 |
| 索引缺失，只能全文搜索 | 效率低，噪音大 |

### 1.2 结构化方案的核心优势

- **按类别组织**：习惯/技能/项目分目录存放，同类信息聚合
- **每条记忆独立文件**：精确读取，不用全量扫描
- **加权二级检索**：总目录粗筛 → 独立文件精读，按权重排序
- **优先级 + 保鲜期**：核心约束永不归档，过时信息自动降级
- **全局索引**：`memory_index.md` 维护所有条目的元数据

## 二、优先级策略详解

### 2.1 三级优先级的设计逻辑

```
high (核心约束)
  ├── 永不自动归档
  ├── 检索时获得最高权重 (+20)
  └── 示例："不要生成图片"、"必须用中文回复"

medium (常用偏好)
  ├── 正常保留，到期后提示确认
  └── 示例："偏好暗色主题"、"用TypeScript"

low (临时信息)
  ├── 90天后纳入归档候选
  └── 示例："本次任务用Tab缩进"、"临时约定"
```

### 2.2 优先级的动态调整

```
新记忆写入 (default: medium)
       │
       ├─ 用户强调? → emphasis=true, upgrade to high
       │
       └─ 发现不再重要? → downgrade to low → 90天后归档候选
```

### 2.3 emphasis 和 mention_count

- **emphasis=true**：用户主动强调/标记重点（"记住这个"、"这个很重要"），检索时获得额外子权重
- **mention_count**：skill 类别的提及次数，>=3 视为"反复提及"，等价于 emphasis 的检索权重

## 三、归档原理

### 3.1 为什么需要归档？

随使用时间增长，记忆文件会不断膨胀。归档解决三个问题：

1. **性能**：保持活跃文件在合理数量内（<=50条/目录）
2. **准确性**：减少过时信息对检索结果的干扰
3. **可追溯性**：归档不等于删除，历史记录仍可查到

### 3.2 归档算法

```
add_memory() 写入新条目
       │
       ▼
该类别目录下 .md 文件数 > 50 ?
       │
      ╱╲
     ╱  ╲ No → 正常结束
    ╱Yes╲
   ╱──────╲
   │      │
   ▼      │
第一轮: priority=low 且 age>90天 → 移入 archive/
   │      │
   ▼      │
仍>50?    │
   │ ╱╲   │
   │╱  ╲  │
   No   Yes│
   │    │  │
   │    ▼  │
   │  第二轮: priority=low 且 age>60天 → 移入 archive/
   │    │  │
   ▼    ▼  ▼
更新 memory_index.md 中的路径
   │
   ▼
返回归档通知
```

### 3.3 归档设计决策

| 选择 | 原因 |
|------|------|
| 只归档 low 优先级 | high 和 medium 仍有价值 |
| 90天阈值 | 约3个月，避免误删又保持精简 |
| 不删除而是移动 | 保留完整历史供必要时检索 |
| 双阈值策略 | 第一轮90天未清理干净则降为60天 |

## 四、加权检索的公平性

不同类别有不同的"公平"定义：

- **project**：最看重时效性——近期任务比旧任务更重要（recency=25 > emphasis=15）
- **habit**：最看重用户强调——明确要求的偏好比时间新旧的更重要（emphasis=25 > recency=15）
- **skill**：最看重反复提及——多次使用的技能比单次记录更有价值（emphasis=30）

这种差异化设计反映了真实的认知优先级：项目的时效性比偏好更强，而偏好中用户明确的要求比时间上的新近更重要。

## 五、工具独占原则

### 为什么禁止裸写文件？

如果允许 Agent 直接写记忆文件，会出现：

1. **索引不一致**：写了文件但忘了更新 `memory_index.md`
2. **格式破坏**：手动拼接 YAML 容易出错
3. **归档绕过**：直接写入导致条目计数不准，归档失效
4. **并发冲突**：多步操作中间被打断导致半更新状态

### cli.py 的保证

```
原子性：单次调用完成全部更新（文件 + 索引同步）
一致性：总目录与文件系统始终同步
完整性：自动触发归档检查
可追溯：所有操作都有 JSON 格式的输入输出
```

## 六、从旧版迁移

如果你从 v1.x（单文件架构：`habits_preferences.md` + `internal_index`）迁移到 v2.x（独立文件架构）：

1. 运行 `python cli.py rebuild --json` 重建索引
2. 旧版单文件中的条目需要手动拆分为独立 `.md` 文件
3. `priority_tags` 和 `internal_index` 字段不再使用，由 `memory_index.md` 的 `entries` 列表替代
4. `emphasis` 和 `mention_count` 是新增字段，旧条目默认值为 `false` 和 `0`

## 七、未来可能的演进方向

| 方向 | 描述 | 复杂度 |

| 向量语义检索 | 将 summary 向量化，支持自然语言语义匹配 | 中 |
| 自动标签推荐 | 基于内容自动生成标签建议 | 低 |
| 记忆冲突检测 | 检测新记忆与已有记忆的矛盾 | 中 |
| 多 Agent 共享 | 支持多个 Agent 实例共享同一记忆库 | 高 |
| MCP 集成 | 将脚本封装为 MCP Server，Skill 框架原生支持工具注册 | 高 |
